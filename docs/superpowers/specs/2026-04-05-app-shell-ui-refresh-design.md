# 应用壳层 UI 优化设计文档

**日期**: 2026-04-05  
**状态**: 设计已确认，待实现

## 1. 背景

当前工具站已经具备基础壳层能力：

- 左侧侧边栏用于按分类切换工具
- 主内容区承载各工具页面
- 已支持侧边栏展开/收起、搜索聚焦和快捷键帮助面板

现状问题不在功能，而在视觉表达：

- 页面整体仍然偏基础 Tailwind 默认风格，产品感不够完整
- 侧边栏、主内容区、按钮、弹层之间的层级关系不够清晰
- 主内容区直接铺在浅灰背景上，当前工具缺少被聚焦的“工作区”感
- 侧边栏是整块深色区域，信息可读性可以更精致，但又不能压过主内容
- 公共按钮和弹层的视觉语言不统一，局部观感偏散

本次优化仅影响应用壳层与公共视觉，不调整各工具的业务逻辑和核心交互。

## 2. 目标

### 2.1 功能目标

- 在不改工具逻辑的前提下，提升应用壳层的整体质感和层级
- 让侧边栏更像导航面板，而不是简单列表
- 让主内容区更像被聚焦的工作台画布
- 统一按钮、浮动入口、快捷键弹层的公共视觉语言
- 保持现有快捷键与侧边栏行为不变

### 2.2 非目标

- 不修改各工具内部业务功能
- 不重做工具页面布局结构
- 不引入新的组件库或设计系统
- 不增加动画驱动的复杂交互
- 不新增主题切换、暗色模式或个性化设置

## 3. 方案对比

### 方案 A：轻量提质

只调整背景、圆角、阴影和边框，让现有结构更精致。

优点：

- 改动最小，风险低
- 不容易影响现有工具页面

缺点：

- 整体观感提升有限
- 无法明显建立“导航层 / 内容层”的工作台结构

### 方案 B：仪表盘壳层

强化侧边栏导航面板、主内容信息头部和悬浮内容区之间的层级关系，形成完整工作台外观。

优点：

- 最符合当前应用“多工具工作台”的定位
- 能在不改业务逻辑的前提下显著提升整体感
- 对现有工具页兼容性较好，主要通过壳层和公共组件生效

缺点：

- 需要同时调整多个公共组件
- 对间距、阴影和背景层级需要统一拿捏，不能只做局部修改

### 方案 C：极简编辑器风格

降低装饰，做更冷静、更高信息密度的专业工具外观。

优点：

- 专注感更强
- 对开发工具语境天然贴合

缺点：

- 与现有偏卡片的工具页风格衔接一般
- 容易显得过于克制，缺少产品识别度

最终采用方案 B：仪表盘壳层。

## 4. 最终设计

### 4.1 整体视觉方向

壳层采用“浅色仪表盘”风格：

- 页面最外层使用柔和浅色渐变和大面积留白，不再是单一灰底
- 侧边栏保留深色系，但改成更轻的半透明导航面板
- 主内容区放入悬浮式内容面板，和背景建立清晰层级
- 全局统一圆角、边框、阴影、背景透明度和模糊感

设计原则：

- 视觉升级优先服务结构清晰，不做为装饰而装饰
- 主内容区永远是视觉焦点，侧边栏只能辅助导航，不能喧宾夺主
- 公共视觉语言统一后，各工具页面应自动受益

### 4.2 侧边栏导航面板

侧边栏维持当前信息结构，但强化导航属性：

- 宽度较当前略增，给搜索框和工具名称更多留白
- 外层改为深色半透明面板，加入柔和边框和阴影
- 顶部标题区除了 `DevTools` 外，增加一行轻量辅助说明
- 搜索框作为导航主入口，背景、边框和聚焦状态更明确
- 分类标题改成更稳定的导航分组样式，图标、文字、箭头统一对齐
- 当前选中工具使用更明确的高亮块、边框或发光感提示
- 快捷键帮助入口保留，但做成辅助按钮样式，而非普通文本链接

不做的事：

- 不改分类逻辑
- 不改搜索行为
- 不增加二级导航或图标栏模式

### 4.3 主内容区结构

主内容区改成“背景画布 + 顶部信息区 + 悬浮内容面板”的三层结构。

背景画布：

- 使用柔和渐变和少量装饰性光斑，提升空间感
- 装饰层只做气氛，不影响阅读和交互

顶部信息区：

- 左侧显示当前工具名称与简短说明
- 右侧保留壳层操作入口，如侧边栏隐藏态的恢复按钮
- 只提供轻量信息提示，不扩展成复杂工具栏

内容面板：

- 将工具页面承载在更明确的主面板中
- 面板头部与内容区分层更清晰，弱化生硬边框
- 窄布局工具页继续受最大宽度限制，宽布局页保持可扩展
- 滚动区域和内边距统一，避免当前不同页面在壳层内显得松散

### 4.4 侧边栏隐藏态入口

当前隐藏态只有一个轻按钮，信息密度偏低。本次改成悬浮控制卡片：

- 固定在主内容区左上角
- 同时显示“菜单”入口和快捷键提示
- 与顶部信息区保持同一视觉体系
- 不覆盖主要内容，不引入复杂悬停交互

目标是降低隐藏侧边栏后“忘记恢复入口”的概率。

### 4.5 快捷键帮助弹层

快捷键帮助仍然是当前页面内专用弹层，但视觉更完整：

- 遮罩层更柔和，突出面板聚焦感
- 面板本体采用更强的层级、圆角和阴影
- 标题、说明、快捷键列表、关闭按钮的间距统一
- 每条快捷键做成更像帮助卡片的样式，而不是简单行布局

交互保持不变：

- 支持点击遮罩关闭
- 支持点击关闭按钮关闭
- 支持 `Esc` 关闭

### 4.6 公共按钮体系

`Button` 组件统一调整视觉节奏：

- 主按钮更饱满，强调色更稳定
- 次按钮采用更轻的边框和更明确的悬停反馈
- 幽灵按钮保持低干扰，但在深浅背景上都要足够可识别
- 统一圆角、聚焦 ring、禁用态和阴影细节

本次只在公共组件收敛样式，不逐个改工具页按钮逻辑。

### 4.7 ToolContainer 容器策略

`ToolContainer` 负责把各工具页面接入新的壳层视觉：

- 标题区做成更像内容卡片头部
- 描述文案层级更清楚
- 内容区域统一留白和滚动策略
- `layout="narrow"` 继续控制最大宽度，但增强页面稳定感

这样可以让现有工具页面在不重写布局的前提下，直接继承新的工作台外观。

## 5. 组件职责边界

- `App`
  - 负责外层背景、整体布局、当前工具信息区和弹层承载
  - 负责隐藏态入口的样式整合
- `Sidebar`
  - 负责导航面板的视觉升级
  - 负责搜索框、分类组、工具项和快捷键入口的统一表现
- `ToolContainer`
  - 负责内容面板的统一结构与留白
- `Button`
  - 负责公共按钮体系收敛
- `index.css`
  - 负责全局背景、基础排版和通用视觉变量

## 6. 风险与控制

主要风险：

- 壳层层级改动过大，导致工具页内部看起来被“套壳”得太重
- 深色侧边栏和浅色主区的对比过强，造成视觉割裂
- 公共按钮样式调整后，部分工具页局部对比度不足
- 内容面板层级增加后，滚动区域或高度计算出现异常

控制策略：

- 只动壳层与公共组件，不修改工具页内部功能结构
- 让主内容区保持阅读优先，避免过度装饰
- 复用现有 `ToolContainer` 宽窄布局模型，减少布局回归风险
- 实现后优先验证带有大文本区、双栏区和切换按钮的典型工具页

## 7. 验证策略

本次以构建和手工回归为主：

- 侧边栏展开/收起行为与快捷键保持正常
- 搜索框聚焦、分类展开和工具切换行为不受影响
- 主内容区在常见工具页中没有出现滚动或高度异常
- 快捷键帮助面板仍支持遮罩关闭、按钮关闭和 `Esc` 关闭
- `Button` 视觉升级后，主按钮、次按钮、幽灵按钮在现有工具页中可读性正常
- 常见页面在桌面宽度和窄窗口下仍保持可用
